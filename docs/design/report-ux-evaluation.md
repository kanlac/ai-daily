# 每日 HTML 报告信息设计、视觉规范与评估量表

## 1. 目标与设计原则

每日简报应从“纯文本摘要”升级为一份可快速决策、可追溯证据、可沉淀历史认知的 HTML briefing。它不是新闻列表，也不是通用 SaaS 仪表盘，而是面向高密度信息消费者与 builder 的「精英情报报告」。

核心目标：

1. **高信噪比**：首屏给出判断，正文保留证据，避免把读者淹没在无差别卡片里。
2. **证据可追溯**：每个判断都能回到原文摘录、中文翻译、链接、时间戳与来源可信度。
3. **跨日可积累**：同一人物、公司、产品、技术趋势、政策事件在多日报告中持续关联，形成记忆层。
4. **桌面优先，移动可读**：桌面提供高密度扫描体验；手机降级为单列、锚点清晰、可快速跳读。
5. **可打印与可归档**：HTML 即阅读界面，也应能输出为干净的 PDF/纸质 briefing。

---

## 2. 信息架构：报告结构

### 2.1 页面总览

建议页面从上到下采用以下结构：

1. Hero / Briefing Header
2. 今日判断 / Executive Judgement
3. 关键变化雷达 / What Changed Today
4. 板块卡片 / Sections
5. 原文引用与中文翻译 / Quotes & Translation
6. 人物背景 / People Context
7. Product Idea Canvas
8. 跨日关联 / Historical Threads
9. 原始来源 Appendix
10. 评估与元数据 / Quality Metadata

### 2.2 Hero / Briefing Header

**目的**：让读者在 10 秒内知道今天报告的主题、优先级与阅读路径。

应包含：

- 报告日期、生成时间、覆盖时间窗。
- 一句话标题：避免泛泛标题，例如“AI 产品生态日报”；应使用判断型标题，例如“开源 Agent 工具链开始从 demo 走向工作流基础设施”。
- 今日主题标签：如 `AI infra`、`consumer app`、`crypto policy`、`hardware`。
- 重要度指示：High / Medium / Low 或 1-5 级，但必须说明原因。
- 阅读预计时间：如 `8 min scan / 25 min deep read`。
- 来源数量、去重后事件数、跨日关联数。
- 顶部操作：搜索、筛选、打印、复制链接、导出 PDF。

推荐布局：

```text
┌──────────────────────────────────────────────┐
│ 日期 + Briefing 编号 + 重要度                │
│ 判断型大标题                                  │
│ 3-5 个今日主题标签                            │
│ sources / events / threads / reading time     │
└──────────────────────────────────────────────┘
```

### 2.3 今日判断 / Executive Judgement

**目的**：先给结论，再给证据。今日判断不应只是摘要，而应回答“这对 builder / investor / operator 意味着什么”。

结构建议：

- **一句话结论**：最重要的判断。
- **3 条核心判断**：每条包含影响、置信度、证据入口。
- **今天不应忽略的弱信号**：1-3 条，不要求确定，但应解释为什么值得跟踪。
- **风险与反例**：避免报告只强化单一叙事。

示例字段：

```text
判断：开发者工具正在从单点插件变成端到端工作流层。
影响：未来 2-4 周应重点观察团队协作、权限、审计日志、部署闭环。
置信度：7/10
证据：来源 #12、#18；历史线索 T-004
```

### 2.4 关键变化雷达 / What Changed Today

用于承接 Hero 和正文，压缩展示“今天新增了什么”。建议使用紧凑列表，而不是大面积卡片。

字段：

- 事件 / 变化
- 变化类型：融资、发布、政策、人物动态、技术突破、价格变化、社区讨论
- 新鲜度：首次出现 / 后续进展 / 反转 / 确认
- 影响范围：个人、公司、行业、监管、开发者生态
- 关联历史：链接到历史 thread

### 2.5 板块卡片 / Section Cards

**目的**：按主题组织高密度信息，但避免“通用 SaaS 卡片堆砌”。

推荐板块：

- AI / Infra
- Consumer / Social
- DevTools / Open Source
- Crypto / Fintech
- Hardware / Robotics
- Policy / Regulation
- Markets / Funding
- Culture / People

每个板块卡片应包含：

1. **板块判断**：这一板块今天的核心变化。
2. **Top events**：最多 3-5 条，按影响力排序，不按抓取顺序排序。
3. **Evidence chips**：来源编号、可信度、时间戳。
4. **Builder implication**：对产品、市场、增长、技术选型的启发。
5. **Open questions**：后续需要验证的问题。

视觉上应更像“投资备忘录 + 工程战情室”，而不是营销网站卡片。卡片之间应有节奏变化：重点板块可横跨双栏；普通板块为紧凑列表；低价值内容收进 appendix。

### 2.6 引用 / 翻译 / 摘录模块

**目的**：保留原文证据，同时降低中文读者理解成本。

每个重要来源至少保留：

- 原文摘录：保持原始语言，避免过度改写。
- 中文翻译：忠实翻译，不添加未在原文出现的信息。
- 解释：说明这段原文为什么重要。
- 来源链接：原始 URL、作者、发布时间、抓取时间。
- 引用级别：核心证据 / 辅助证据 / 背景资料。

推荐展示：

```text
[原文] expandable block
“...”

[中文翻译]
“...”

[为什么重要]
这说明……，与昨日 T-003 中的……形成呼应。
```

默认状态：正文只显示中文翻译与一句解释；原文折叠。点击“展开原文”后显示完整摘录。

### 2.7 人物背景 / People Context

**目的**：当报告提到人物时，读者不应需要额外搜索才能理解其重要性。

人物模块包含：

- 姓名、头像或简短身份标签。
- 当前角色、过往关键履历。
- 与今日事件的关系。
- 历史出现次数与最近出现日期。
- 可信度提醒：人物观点是事实、立场、营销还是猜测。
- 相关实体：公司、项目、投资机构、社区。

示例：

```text
人物：Jane Doe
身份：前 OpenAI researcher，现任某 AI infra startup 创始人
为什么重要：其团队近期连续发布 agent runtime 相关工具，今日融资消息可能意味着该方向资本加速。
历史关联：2026-05-12 T-011；2026-05-15 T-019
```

### 2.8 Product Idea Canvas

**目的**：把信息转化为可执行产品想法，而不是停留在资讯消费。

每条 idea 应结构化为：

- Idea 名称
- 触发来源：对应事件、引用、历史 thread
- 用户痛点
- 目标用户
- 现有替代方案
- 差异化机会
- MVP 形态
- Distribution / GTM 假设
- 风险与反证
- 下一步验证动作
- 优先级评分：Impact、Feasibility、Timing、Evidence strength

建议每日报告只保留 1-3 条高质量 idea，避免把所有新闻都硬转成 idea。

### 2.9 跨日关联 / Historical Threads

**目的**：让日报成为连续情报系统，而非一次性文档。

每条跨日关联包含：

- Thread ID：如 `T-2026-05-agent-infra-001`
- 主题名称
- 首次出现日期
- 最近更新日期
- 今日新增证据
- 历史摘要：3-5 行
- 趋势状态：升温 / 降温 / 反转 / 确认 / 噪音
- 相关报告链接
- 下一次观察点

在正文中所有相关事件都应链接到 thread；thread 页面或区块应反向列出相关来源。

### 2.10 原始来源 Appendix

**目的**：完整保留来源，满足审计、复核和深读需求，但不干扰主线阅读。

Appendix 字段：

- 来源编号：`S-001`
- 标题
- URL
- 作者 / 发布方
- 发布时间
- 抓取时间
- 来源类型：官方公告、社交媒体、新闻媒体、论文、代码仓库、论坛等
- 可信度等级
- 使用位置：在哪些判断、板块、idea、thread 中被引用
- 原始摘录
- 中文翻译
- 备注：是否有重复、冲突、待验证信息

Appendix 默认折叠，可搜索，可按来源类型、可信度、主题筛选。

## 3. 视觉风格规范

### 3.1 总体气质

视觉关键词：

- **Elite builder briefing**
- **Dense but calm**
- **Editorial intelligence**
- **Evidence-first**
- **Desk research meets product strategy**

避免：

- 通用 SaaS dashboard 的浅灰背景 + 白卡片无限堆叠。
- 过度圆角、过度阴影、空洞渐变。
- 用颜色装饰代替信息层级。
- 新闻门户式标题党与广告位感。

### 3.2 桌面优先布局

推荐桌面宽度：`1200px - 1440px` 内容容器。

布局建议：

```text
┌──────────────┬────────────────────────────┬──────────────┐
│ 左侧导航     │ 主内容                     │ 右侧情报栏    │
│ anchors      │ report body                │ threads/meta  │
└──────────────┴────────────────────────────┴──────────────┘
```

- 左侧：锚点导航、板块跳转、阅读进度。
- 中间：主报告正文，宽度约 `720px - 860px`。
- 右侧：今日指标、相关历史、人物卡、来源可信度、操作按钮。
- Hero 可跨全宽，正文后进入三栏结构。

### 3.3 字体与排版

建议：

- 中文正文：系统字体优先，如 `-apple-system, BlinkMacSystemFont, "PingFang SC", "Noto Sans CJK SC", sans-serif`。
- 英文摘录 / 代码 / URL：`ui-monospace, SFMono-Regular, Menlo, Consolas, monospace`。
- 标题：使用稍紧字距、较高字重，但不夸张。
- 正文行高：中文 `1.65 - 1.8`。
- 高密度列表行高：`1.35 - 1.5`。
- 数字、评分、来源编号使用等宽或 tabular nums。

字号建议：

| 元素 | 桌面 | 移动 |
| --- | ---: | ---: |
| Hero 标题 | 32-42px | 24-30px |
| 一级标题 | 24-28px | 22-24px |
| 二级标题 | 18-22px | 18-20px |
| 正文 | 15-17px | 15-16px |
| 元信息 | 12-13px | 12px |
| 原文摘录 | 13-15px | 13-14px |

### 3.4 色彩系统

建议采用低饱和、偏 editorial 的色彩：

- 背景：暖白或深墨灰，避免纯白刺眼。
- 主文字：接近黑色但不纯黑。
- 次级文字：中性灰。
- 强调色：单一主强调色，如深蓝、暗金、铜绿或电光蓝。
- 状态色：只用于表达语义，如风险、确认、反转、升温。

颜色使用原则：

- 颜色表达信息，不做无意义装饰。
- 同一语义全站一致，例如红色只表示风险/冲突，不表示“热门”。
- 来源可信度、趋势状态、重要度使用形状 + 文字 + 颜色三重编码，避免仅靠颜色。

### 3.5 组件规范

#### Evidence Chip

用于标记来源与证据强度。

格式：`S-012 · 官方 · 可信度 8 · 2h ago`

#### Judgement Block

用于核心判断。

必须包含：判断、证据、置信度、影响、反例。

#### Quote Block

区分原文、翻译、解释。

- 原文：等宽或衬线感样式，折叠显示。
- 翻译：正文样式，高可读。
- 解释：使用边注或淡色背景。

#### Thread Link

格式：`T-004 · Agent runtime consolidation · 升温`

#### Idea Canvas

应像一页产品备忘录，而不是普通 bullet list。可使用 2 列：左侧用户与痛点，右侧 MVP 与验证。

### 3.6 移动端断点

推荐断点：

- `>= 1200px`：三栏布局。
- `900px - 1199px`：主内容 + 折叠右侧栏；左侧导航变为顶部 sticky。
- `600px - 899px`：单列布局；Hero 压缩；板块导航横向滚动。
- `< 600px`：纯单列；来源、原文、历史默认折叠；底部固定操作栏可选。

移动端要求：

- 首屏必须仍能看到标题、日期、重要度、核心判断入口。
- 每个板块标题下方显示 1 行 summary，展开后才显示完整证据。
- 原文摘录默认折叠，避免长英文段落占满屏幕。
- 锚点导航应变为顶部 sticky tabs 或目录抽屉。
- 点击目标高度不小于 `44px`。

### 3.7 可打印规范

打印/PDF 样式应独立优化：

- 隐藏搜索框、筛选器、展开按钮、悬浮导航。
- 展开关键原文摘录，但不展开所有 appendix。
- 保留 URL 或脚注编号，保证离线可追溯。
- 避免卡片在分页处被截断：使用 `break-inside: avoid`。
- 使用白底黑字，减少背景色与阴影。
- 页面顶部保留日期、标题、页码。

## 4. 交互设计

### 4.1 搜索与筛选

搜索范围：

- 标题
- 正文判断
- 原文摘录
- 中文翻译
- 人物
- 公司 / 产品 / 项目
- 来源 URL
- Thread ID

筛选维度：

- 板块
- 重要度
- 来源类型
- 可信度
- 趋势状态
- 是否含 product idea
- 是否有跨日关联
- 是否为首次出现

搜索结果应高亮命中词，并显示结果所在上下文与锚点。

### 4.2 展开原文

展开行为：

- 默认显示中文翻译 + 解释。
- 点击“展开原文”显示原文摘录、来源元数据、复制引用按钮。
- 支持“展开本节全部原文”和“全部收起”。
- 对特别长的原文应二级折叠：先显示关键段，再显示完整摘录。

### 4.3 锚点导航

导航项：

- 今日判断
- 关键变化
- 各主题板块
- 人物背景
- Idea Canvas
- 跨日关联
- Appendix

行为：

- 桌面左侧 sticky。
- 移动端顶部 sticky tabs 或目录抽屉。
- 当前阅读位置高亮。
- URL hash 可复制分享。

### 4.4 相关历史链接

所有具有跨日意义的事件都应显示：

- 相关 thread
- 上一次出现日期
- 历史判断是否被证实或推翻
- 相似事件
- 同一人物 / 公司 / 产品的历史记录

链接样式应区分：

- `继续升温`：历史趋势被加强。
- `反转`：与历史判断冲突。
- `确认`：之前弱信号被证实。
- `噪音`：历史上多次出现但未形成实质变化。

### 4.5 低干扰辅助交互

- 复制当前段落链接。
- 复制证据引用。
- 标记稍后阅读。
- 一键只看高优先级。
- 一键只看 product ideas。
- 键盘快捷键：`/` 搜索，`j/k` 上下事件，`o` 展开原文，`p` 打印。

## 5. 评分 Rubrics

总分建议为 70 分，7 个维度各 10 分。通过阈值建议：

- 单项最低通过：`>= 7`
- 总分通过：`>= 52/70`
- 发布优秀线：`>= 60/70`
- 若“证据可追溯”或“时效性”低于 7，即使总分达标也不应自动发布。

### 5.1 美观 / Visual Quality

| 分数 | 标准 |
| ---: | --- |
| 1-3 | 像未设计的 HTML 或堆叠卡片；层级混乱；颜色随机。 |
| 4-6 | 基础可读，但缺少 editorial 气质；组件重复单调；重点不突出。 |
| 7-8 | 视觉层级清晰，有 builder briefing 气质，信息密度与留白平衡。 |
| 9-10 | 形成独特视觉语言；高密度但优雅；桌面、移动、打印均稳定。 |

通过阈值：`>= 7`

### 5.2 信噪比 / Signal-to-Noise Ratio

| 分数 | 标准 |
| ---: | --- |
| 1-3 | 大量低价值新闻罗列；没有排序；判断缺失。 |
| 4-6 | 有摘要，但重要信息与背景噪音混杂；重复较多。 |
| 7-8 | 重要内容优先；低价值信息进入 appendix；每节有明确判断。 |
| 9-10 | 报告像经过资深分析师编辑；每个元素都有决策价值。 |

通过阈值：`>= 7`

### 5.3 阅读体验 / Reading Experience

| 分数 | 标准 |
| ---: | --- |
| 1-3 | 首屏无重点；长段落难扫读；读者无法快速定位。 |
| 4-6 | 可读但节奏单一；锚点、折叠、摘要不足。 |
| 7-8 | 可 5 分钟扫读，也可 30 分钟深读；导航和节奏良好。 |
| 9-10 | 阅读路径极清晰；不同读者都能快速进入自己的关注层级。 |

通过阈值：`>= 7`

### 5.4 时效性 / Timeliness

| 分数 | 标准 |
| ---: | --- |
| 1-3 | 明显过时；缺少发布时间；无法区分今日新增与旧闻。 |
| 4-6 | 有时间信息，但新旧变化表达不清。 |
| 7-8 | 清楚标注时间窗、抓取时间、首次出现/后续进展。 |
| 9-10 | 能识别反转、确认、升温等动态变化，并突出真正的新信息。 |

通过阈值：`>= 7`；低于 7 不自动发布。

### 5.5 证据可追溯 / Evidence Traceability

| 分数 | 标准 |
| ---: | --- |
| 1-3 | 判断无来源；链接缺失；原文不可查。 |
| 4-6 | 有链接但证据与判断关系弱；原文/翻译不完整。 |
| 7-8 | 每个核心判断都有来源编号、原文摘录、翻译和可信度。 |
| 9-10 | 可从任意判断一跳到原文、历史、冲突证据与 appendix。 |

通过阈值：`>= 7`；低于 7 不自动发布。

### 5.6 去重与关联 / Deduplication & Association

| 分数 | 标准 |
| ---: | --- |
| 1-3 | 同一事件多次出现；人物/公司/主题没有关联。 |
| 4-6 | 基础去重，但跨日记忆弱；相似事件未合并。 |
| 7-8 | 同源/转载/相似事件被合并；有 thread 和历史链接。 |
| 9-10 | 能识别趋势、反转、确认、噪音，并自动更新历史线索。 |

通过阈值：`>= 7`

### 5.7 移动端体验 / Mobile Experience

| 分数 | 标准 |
| ---: | --- |
| 1-3 | 横向溢出；字体过小；导航不可用。 |
| 4-6 | 单列可读，但信息过长；折叠与锚点不足。 |
| 7-8 | 移动端首屏清晰；板块可跳转；原文与 appendix 默认折叠。 |
| 9-10 | 手机上也有优秀扫读体验；触控、导航、搜索、打印/分享稳定。 |

通过阈值：`>= 7`

## 6. 自动评估与 5-10 轮改进循环

### 6.1 每轮循环输入

每轮生成后收集：

- HTML 渲染截图：桌面、平板、手机。
- 报告 DOM 结构与可访问性结果。
- 数据统计：来源数、事件数、重复率、原文覆盖率、翻译覆盖率、thread 覆盖率。
- Rubric 评分：7 个维度各 1-10。
- 失败项说明：每个低于阈值的维度给出具体证据。

### 6.2 自动改进策略

#### 若美观 < 7

- 减少同质卡片数量，改用 editorial section、时间线、证据表格混排。
- 统一字号、间距、颜色 token。
- 提升 Hero 与核心判断区视觉权重。
- 降低阴影、渐变、随机强调色。

#### 若信噪比 < 7

- 将低可信、低影响内容移动到 appendix。
- 每个板块强制生成一句判断；没有判断的事件降级。
- 合并重复事件与转载源。
- 限制每板块 Top events 数量。

#### 若阅读体验 < 7

- 增加锚点导航与阅读进度。
- 将长段落拆为判断、证据、影响、反例。
- 原文默认折叠，保留一键展开。
- 增加摘要层：Hero summary、section summary、event summary。

#### 若时效性 < 7

- 补齐发布时间、抓取时间、首次出现时间。
- 明确标记“今日新增 / 后续进展 / 旧闻背景”。
- 对超过时间窗的来源降权或移入背景区。
- 提高反转、确认、升温事件的排序权重。

#### 若证据可追溯 < 7

- 强制所有核心判断绑定至少 2 个来源或 1 个高可信官方来源。
- 为每个来源生成 `S-ID`，为每个 thread 生成 `T-ID`。
- 检查每个引用是否有 URL、原文摘录、中文翻译。
- 对无法追溯的判断添加 `needs verification` 或删除。

#### 若去重/关联 < 7

- 基于 URL canonical、标题相似度、实体识别、时间窗口合并重复事件。
- 为人物、公司、产品建立实体 key。
- 自动查找过去 7/14/30 天相关 thread。
- 标记趋势状态：升温、降温、反转、确认、噪音。

#### 若移动端 < 7

- 强制单列布局。
- 隐藏右侧栏，改为可展开模块。
- 原文、appendix、长表格默认折叠。
- 检查横向溢出、触控目标、首屏信息完整性。

### 6.3 5-10 轮执行框架

建议最多执行 10 轮，通常 5-7 轮应达到可发布质量。

| 轮次 | 目标 | 自动动作 | 停止条件 |
| ---: | --- | --- | --- |
| 1 | 生成基础 HTML | 完成结构、来源、翻译、appendix | DOM 完整，无致命缺块 |
| 2 | 提升信息层级 | 重排 Hero、今日判断、板块顺序 | 信噪比 >= 7 |
| 3 | 补强证据 | 补来源编号、原文、翻译、链接 | 证据可追溯 >= 7 |
| 4 | 做去重与跨日关联 | 合并重复事件，写入 thread | 去重/关联 >= 7 |
| 5 | 优化视觉 | 调整布局、typography、组件节奏 | 美观 >= 7 |
| 6 | 优化阅读体验 | 增加锚点、折叠、摘要层 | 阅读体验 >= 7 |
| 7 | 优化移动端 | 检查断点、溢出、触控 | 移动端 >= 7 |
| 8 | 优化打印与归档 | print CSS、URL footnotes | PDF 预览可读 |
| 9 | 复核时效性 | 标记今日新增与旧闻背景 | 时效性 >= 7 |
| 10 | 最终发布判定 | 综合评分与失败项复查 | 总分 >= 52 且关键项通过 |

### 6.4 自动发布门禁

报告只有在满足以下条件时才可发布：

- 总分 `>= 52/70`。
- 美观、信噪比、阅读体验、去重/关联、移动端均 `>= 7`。
- 时效性、证据可追溯均 `>= 7`，否则强制阻断。
- 每个核心判断至少有可点击来源。
- Appendix 中所有来源有 URL 或明确说明来源不可公开。
- 移动端无横向滚动。
- 打印预览没有重大截断。

### 6.5 自动改进优先级

若多个维度不达标，按以下顺序修复：

1. 证据可追溯
2. 时效性
3. 去重与关联
4. 信噪比
5. 阅读体验
6. 移动端
7. 美观

原因：证据与时效性决定报告是否可信；去重和信噪比决定是否值得读；阅读与视觉决定是否能长期使用。

## 7. HTML 实现建议

### 7.1 语义结构

使用语义化 HTML：

- `<header>`：Hero。
- `<nav>`：锚点导航。
- `<main>`：主报告。
- `<section>`：板块。
- `<article>`：事件或 idea。
- `<aside>`：人物、历史、来源元数据。
- `<details>` / `<summary>`：原文、appendix、移动端折叠。
- `<footer>`：生成元数据与评估分。

### 7.2 数据属性

为交互预留 data attributes：

```html
<article
  data-section="ai-infra"
  data-importance="high"
  data-source-ids="S-012 S-018"
  data-thread-ids="T-004"
  data-entities="OpenAI, Anthropic, agent runtime"
>
```

这能支持搜索、筛选、复制引用、历史链接和自动评分。

### 7.3 可访问性

- 所有交互按钮必须可键盘操作。
- 折叠区使用原生 `<details>` 或补齐 ARIA 状态。
- 颜色对比度至少满足 WCAG AA。
- 图标必须有文字标签或 `aria-label`。
- 锚点跳转后焦点应移动到目标区域。

## 8. 发布前检查清单

- [ ] Hero 是否在首屏表达今天最重要判断？
- [ ] 今日判断是否包含证据、置信度、反例？
- [ ] 每个板块是否有自己的判断，而不是新闻堆叠？
- [ ] 核心引用是否有原文、中文翻译、链接？
- [ ] 人物是否有背景与历史关联？
- [ ] Product idea 是否有痛点、MVP、GTM、风险与验证动作？
- [ ] 跨日 thread 是否被正确链接？
- [ ] Appendix 是否完整且可搜索？
- [ ] 桌面是否高密度但不拥挤？
- [ ] 手机是否无横向滚动、首屏可读？
- [ ] 打印/PDF 是否保留标题、日期、来源与页码？
- [ ] Rubric 是否总分达标且关键项通过？

## 9. 最重要的 3 条设计原则

1. **先判断，后证据，证据可回溯**：读者先获得结论，但每个结论都能一跳回到原文、翻译、来源和历史线索。
2. **高密度不等于卡片堆砌**：用 editorial 信息层级、节奏变化和可折叠证据承载复杂内容，而不是无限 SaaS 卡片。
3. **日报必须长出记忆**：人物、产品、公司、趋势和 idea 要跨日关联，持续更新为 thread，形成可积累的情报系统。
